# JNTO 品質評価パイプライン — 問題点一覧

## 概要

テクスチャ最適化の品質判定パイプラインに、判定結果の妥当性を根本的に損なう致命的欠陥が複数存在する。
本ドキュメントは発見された全問題を深刻度順に整理し、あるべき姿を示す。

---

## Critical — アルゴリズムが根本的に誤った結果を出す

### C1. ダウンスケール劣化が評価されていない

**箇所:** `Phase2Pipeline.cs:47-53`

**現状:**
```
original → resize(size) → [A: 非圧縮]
original → resize(size) → compress → decompress → [B: 圧縮済]
比較: A vs B  ← 圧縮アーティファクトのみ
```

`originalForCompare` と `resized` が両方とも `ResolutionReducer.Resize(original, size)` で同サイズに縮小されており、品質ゲートは「同サイズでの非圧縮 vs 圧縮」しか比較していない。

**何が起きるか:** 4K→256に縮小しても、256同士の比較なので模様消失・ぼやけが一切検出されない。どんなに攻撃的な縮小でも圧縮アーティファクトが小さければPassする。

**重要度:** ダウンスケールによるディテール消失は圧縮アーティファクトよりも圧倒的に影響が大きい。テクスチャサイズを半減するだけで高周波情報の大半が失われるのに対し、BC7/DXT圧縮のアーティファクトは相対的に軽微。現在のパイプラインはこの主要な劣化要因を完全に無視しており、品質ゲートとして機能していない。

**あるべき姿:** 以下の2つのアプローチのいずれかまたは両方:

1. **アップスケール比較:** 候補をoriginalサイズにアップスケールして直接比較
```
original → [A: 基準 (フルサイズ)]
original → resize(size) → compress → decompress → upscale(originalSize) → [B: 最終結果]
比較: A vs B  ← ダウンスケール劣化 + 圧縮アーティファクトを同時評価
```

2. **ダウンスケール品質の独立評価メトリクス:** リサイズ前後の高周波エネルギー保存度、パターン構造保持度などをサイズ非依存で評価するメトリクスを用いる（C3のメッシュ密度評価との統合が自然）

---

### C2. ノーマルマップが色画像として評価されている

**箇所:** `FlipMetric.cs:78-89`, `ChromaDriftMetric.cs:30-35`

**現状:** FLIP・ChromaDrift共にsRGBガンマ→リニア変換、XYZ→YCxCz/Lab色空間で比較している。

**問題:** ノーマルマップのピクセル値は法線方向ベクトル (tangent space) であり、色ではない。
- sRGBガンマ変換は法線値に対して非線形な歪みを加える
- Lab色差は法線ベクトルの角度差と無関係
- 布の織り目の質的変化（ざらざら→つるつる）は法線の局所分散変化であり、色差では検出不可能

**あるべき姿:** ノーマルマップ専用メトリクスを用いる。
- 法線をデコード: `n = normalize(r*2-1, g*2-1, sqrt(1 - x² - y²))`
- **角度差メトリクス:** `acos(dot(n_orig, n_candidate))` の統計量
- **局所分散保存メトリクス:** 局所ウィンドウ内の法線方向分散が保存されているか（布の織り目検出）
- **レンダリング比較 (オプション):** 複数照明方向でレンダリングした結果をFLIPで比較

---

### C3. メッシュ上の実視覚サイズで品質評価していない

**箇所:** `Phase2Pipeline.cs`, `DegradationGate.cs`

**現状:** テクスチャ全体を均一にピクセル単位で評価している。メッシュへの割り当て（UV密度）を品質評価に反映していない。

**問題:** テクスチャはいたるところ模様が異なる。同一テクスチャ内でもUV領域ごとにメッシュ上での実視覚サイズ（テクセル密度: world面積/UV面積）が大きく異なり、各領域のテクスチャ内容も異なる。
- 顔に小さいUV島で割り当てられた領域 → テクセル密度が低い → 1ピクセルの劣化が大きな面積に影響 → 劣化が目立つ
- 服の裏側に大きいUV島で割り当てられた領域 → テクセル密度が高い → 劣化が目立たない
- さらに各領域の模様の複雑さが異なるため、同じ縮小率でも影響度が違う

現在これらが等価に扱われており、重要な領域の劣化が平均に埋もれる。

**あるべき姿:** 全ての三角形で品質を評価し、最も劣化影響が大きい箇所を基準とする。

具体的には:
1. 各三角形について world面積/UV面積 比から実効テクセル密度を算出
2. 各三角形のUV領域に対して、縮小前後のテクスチャ内容をその三角形の実効表示サイズで品質評価
3. テクスチャ全体の合否は**全三角形中の最悪スコア**で判定（模様消失やぼやけの影響が最も大きい箇所が基準）
4. 一つのテクスチャが複数Rendererのサブメッシュで共有される場合、全サブメッシュの全三角形にわたる最悪値を使う

**補足:** `DensityCalculator` はメッシュ密度からターゲットサイズを算出しており、ここまでは正しい。問題はその後の品質ゲートがメッシュ密度情報を全く使わずテクスチャを均一評価している点。

### C4. SsimMetricがグローバル計算かつ位置別の実効縮小率を考慮していない

**箇所:** `SsimMetric.cs:14-29`

**現状:** 画像全体の平均・分散を使った単一SSIM値を計算。

**問題:**
1. **グローバル計算:** 標準SSIMは11×11ガウシアンウィンドウで局所計算し全ウィンドウの平均を取る。画像全体のグローバル統計量は単なる相関係数に退化し、構造的類似度としての意味を持たない。ぼやけた画像でもグローバル統計量が近ければ高スコアが出る。
2. **位置無関係:** C3で述べたように、テクスチャの各位置はメッシュ上で異なる実効縮小率を持つ。SSIMを含む全メトリクスは、各位置の実効縮小率（テクセル密度に基づく）を考慮した評価方式にすべき。画像全体を均一に評価するのではなく、三角形ごとの実効表示サイズでの局所評価が必要。

**あるべき姿:** Wang et al. 2004 の標準ウィンドウSSIM実装に修正し、C3の三角形ごとの評価フレームワークに統合する。さらに可能であれば MS-SSIM (Multi-Scale SSIM) に拡張。

## High — 判定精度に重大な影響

### H1. ChromaDriftが最大値1ピクセルで判定

**箇所:** `ChromaDriftMetric.cs:21`

**現状:** 全ピクセル中の単一最大ΔE00で判定: `if (de > maxDe) maxDe = de;`

**問題:** ノイズや極小領域の異常値1ピクセルでメトリクス全体が支配される。特に圧縮後は個別ピクセルに大きな偏差が出やすく、知覚的に問題ない画像が不合格になりうる。

**あるべき姿:** 99thパーセンタイル、または上位1%の平均を使用。

---

### H2. 実装済みメトリクスが未使用、最適な選択がされていない

**箇所:** `DegradationGate.cs:12-15`

以下が実装済みだが `DegradationGate` に組み込まれていない:

| メトリクス | 検出対象 | 対応する要件 |
|---|---|---|
| **BandingMetric** | グラデーション階段状アーティファクト | BC7→DXT変換時のバンディング |
| **BlockBoundaryMetric** | 4pxブロック境界の不連続 | DXT圧縮の典型的アーティファクト |
| **HighFrequencyMetric** | 高周波ディテール消失 | 細かい模様のぼやけ、線状パターンの劣化 |
| **RingingMetric** | エッジ近傍のリンギング | 圧縮によるエッジ周辺のアーティファクト |
| **SsimMetric** | 構造的類似度 | (C4修正後に有効化すべき) |

**問題:** 全メトリクスを同時に有効化すべきという意味ではない。各劣化タイプに対して最も検出精度が高いメトリクスが選択されるべき。メトリクス間には検出対象の重複があり（例: FLIPの特徴パイプラインとHighFrequencyMetricはどちらも高周波を扱う）、冗長な評価は計算コストを増やすだけで精度向上に寄与しない場合がある。

**あるべき姿:** 各劣化タイプ（ダウンスケールぼやけ、圧縮バンディング、ブロック境界、色ドリフト等）に対して、最も効果的な1つのメトリクスを選定する。H3のロール別メトリクスセットと合わせて、テクスチャタイプ×劣化タイプの組み合わせで最適なメトリクスを割り当てる設計とする。

---

### H3. テクスチャロール別のメトリクスセットがない

**箇所:** `DegradationGate.cs`

**現状:** 全テクスチャタイプ (ColorOpaque, NormalMap, SingleChannel, MatCapOrLut) に同一の3メトリクスを適用。

**問題:**
- NormalMapにFLIPのsRGB色差 → 無意味 (C2参照)
- SingleChannelにChromaDrift → 色差は1チャンネルでは過剰
- MatCapOrLutにはBandingが特に重要だが未適用

**あるべき姿:**

| ロール | 必須メトリクス | 追加候補 |
|---|---|---|
| NormalMap | 法線角度差, 局所分散保存 | レンダリング比較 |
| ColorOpaque | FLIP, ChromaDrift, HighFrequency, Banding, BlockBoundary | MS-SSIM |
| ColorAlpha | FLIP, ChromaDrift, AlphaQuantization, HighFrequency, Banding | — |
| SingleChannel | Windowed SSIM, Banding | — |
| MatCapOrLut | ChromaDrift, Banding | — |

---

### H4. リサイズが正方形前提

**箇所:** `ResolutionReducer.cs:9`

**現状:** `RenderTexture.GetTemporary(targetSize, targetSize, ...)` — 常に正方形。

**問題:** 元テクスチャが非正方形 (例: 2048×1024) でもアスペクト比を無視して正方形にリサイズされ、テクスチャが歪む。

**あるべき姿:** 元テクスチャのアスペクト比を保持してリサイズ。長辺を `targetSize` にし、短辺を比率に応じて決定。

---

## Medium — 特定条件で誤判定

### M1. BlockBoundaryMetricが元テクスチャを無視

**箇所:** `BlockBoundaryMetric.cs:13`

**現状:** `candidate` のみ読み取り、`original` パラメータを完全に無視。

**問題:** 元テクスチャに4px整列のグリッド模様やタイルパターンがある場合、圧縮アーティファクトではないのに偽陽性になる。

**あるべき姿:** originalの4pxグリッド上の差分比率を差し引いてから判定。

---

### M2. BandingMetricの正規化が恣意的

**箇所:** `BandingMetric.cs:52`

**現状:** `return Mathf.Clamp01(top / (samples * 0.02f));` — `0.02f` の根拠が不明。

**あるべき姿:** 実テクスチャでの校正データに基づく正規化係数、またはサンプル数非依存の統計的指標 (例: χ²検定ベースの一様性スコア) を使用。

---

### M3. HighFrequencyMetricが候補自身のブラーを基準に使う

**箇所:** `HighFrequencyMetric.cs:33-34`

**現状:**
```csharp
float loR = Lum(po[idx]) - blurO[idx];  // original - blur(original)
float lcR = Lum(pc[idx]) - blurC[idx];  // candidate - blur(candidate)
```

**問題:** 候補のブラー `blurC` は候補自身から計算される。圧縮アーティファクト（リンギング等）が高周波成分を持つため、ディテール消失していてもアーティファクトの高周波で「保存されている」と誤判定する可能性がある。

**あるべき姿:** `blurO` のみを基準に使い、`lcR = Lum(pc[idx]) - blurO[idx]` とする。または元テクスチャの高周波成分と候補の高周波成分の相関係数で評価。

---

### M4. mipレベル0のみの評価

**現状:** メトリクスは全て `GetPixels()` で mip 0 のみ読み取り。

**問題:** 3Dレンダリングではmipチェーン全体が使われる。圧縮フォーマットが低mipレベルで特にアーティファクトを生じるケースがある（DXT1の4×4ブロック制約は低解像度mipで影響が大きい）。

---

## Low — 設計上の制約

### L1. FLIPのPPDが固定

**箇所:** `FlipMetric.cs:20` — `const float DefaultPpd = 60.5f;`

DensityCalculatorのviewDistanceと連動し、遠距離で使われるテクスチャには低PPDを適用すべき。

### L2. テクスチャごとの閾値オーバーライドがない

全閾値が `DegradationThresholds.cs` にハードコード。特定テクスチャ（顔など）だけ厳しくする手段がない。

### L3. FlipMetricのコメントが実態と乖離

`FlipMetric.cs:9` のコメント「Banding / BlockBoundary / Ringing / HighFrequency / SSIM を1つに代替」は事実に反する。FLIPはこれらの特化メトリクスを代替しない。FLIPの特徴パイプラインはエッジ/ポイントを検出するが、バンディングやブロック境界アーティファクトの検出は設計目的外。

## 理想的なパイプラインの全体像

```
入力: original (フルサイズテクスチャ), targetSize, TextureRole,
      triangles[] (各三角形のUV座標 + world面積/UV面積比)

1. テクスチャロールに応じた最適メトリクスを選択:
   - NormalMap → 法線角度差, 法線局所分散保存
   - Color    → ダウンスケールぼやけに最適なメトリクス + 圧縮アーティファクトに最適なメトリクス
   - etc.

2. サイズ探索ループ (targetSize → originalSize, ×2ステップ):
   a. candidate = resize(original, size) → compress(fmt) → decompress

   b. 三角形ごとの品質評価:
      各三角形について:
        - 三角形のUV領域を特定
        - その三角形の実効テクセル密度 (worldArea/uvArea) から実効表示サイズを算出
        - originalとcandidateの当該UV領域を実効表示サイズで比較
        - 三角形ごとの劣化スコアを記録
      テクスチャ全体のスコア = 全三角形中の最悪スコア

   c. 最悪スコアが閾値以内 → (size, fmt) を返す

3. 全サイズ・フォーマット不合格 → 元テクスチャを返す
```

**C1との統合:** 三角形ごとの評価はoriginalとcandidateを実効表示サイズで比較するため、
ダウンスケール劣化と圧縮アーティファクトの両方が自然に評価される。
アップスケール比較は不要となり、実際の見え方に直結した評価が可能になる。

## 優先度順の対応方針

| 順序 | 項目 | 影響 |
|---|---|---|
| 1 | C1: 比較基準をoriginalにする | パイプラインの根幹修正 |
| 2 | C3: メッシュ実視覚サイズでの評価 | 品質判定の妥当性確保 |
| 3 | C2: ノーマルマップ専用メトリクス | 現状ノーマルマップが全く評価できていない |
| 4 | H2+H3: 未使用メトリクス有効化 + ロール別選択 | バンディング・ディテール消失の検出 |
| 5 | C4: SsimMetricのウィンドウ化 | 有効化の前提条件 |
| 6 | H1: ChromaDriftのパーセンタイル化 | 偽不合格の排除 |
| 7 | H4: 非正方形テクスチャ対応 | エッジケースの修正 |
| 8 | M1-M4: 中程度の問題群 | 精度向上 |
| 9 | L1-L3: 設計改善 | 長期的品質 |
